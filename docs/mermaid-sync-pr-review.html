01

The problem

A trust gap, not a tooling gap.

Diagrams rot. Code moves on; the architecture diagram in the repo quietly stops being true. The Mermaid Diagram Sync bot (a GitHub App) fixes the rot by regenerating connected diagrams whenever the source they describe changes — automatically, on every PR.

But an automated edit you can't see is an automated edit you can't trust. When the bot rewrites a .mmd file, three things go wrong at once:

  • The change is invisible — it lands as just another file in the diff, indistinguishable from a human edit.
  • The change is unreviewable — Mermaid re-lays-out on regeneration, so a one-node change can look like the whole diagram moved. A raw text diff reads as alarming noise.
  • In a real PR the bot touches many diagrams at once — so the developer would have to open each file one by one just to find what changed.
Old problem"How do we keep diagrams in sync with code?"
Real problem"How do we make a bot's edit trustworthy enough to accept?"
02

Why we started

The bot shipped first; the trust layer didn't.

The Mermaid Diagram Sync GitHub App can already keep diagrams current. What it can't do is make a developer comfortable with that. Auto-regeneration without review is the kind of feature people turn off — one surprising diff and the bot reads as "the thing that messes with my files."

So the bet is that the review experience, not the regeneration, is what makes the bot adoptable. If a developer can glance at what changed, understand it in seconds, and accept it with one click — inside VS Code, without context-switching to the PR — then automatic sync becomes a feature you leave on.

Framing. This is a trust interface for an existing automation, not a new automation. Every slice is measured by one question: does it make the next bot edit easier to accept?
03

The thesis & system

Ship trust in vertical slices — each one usable, each one proving the next.

Rather than build the full "review cockpit" up front, the work is sliced so every increment is shippable on its own and earns the right to the next. Acknowledge a change before you diff it; diff one before you aggregate many; aggregate before you handle the hard edge cases.

Two principles hold the roadmap together

  • Never gate the moment of value. Reviewing a single bot edit is always free and works logged-out. Only the aggregation and convenience — the multi-diagram list and bulk accept — sit behind login.
  • Progressive disclosure. The single review surface already tested as "heavy to use"; stacking many of them compounds it. Each slice has to reduce load, not add it.
Login is an activation lever, not a paywall. 96% of activated users never log in. The multi-diagram cockpit is a concrete reason to create an account — without ever paywalling the act of reviewing. A paid tier on top is a separate, later decision.
04

The slice roadmap

Ten slices, from "acknowledge it changed" to "fix it before the PR opens."

Three slices shipped — acknowledge → diff → act. After the June analytics (§08), the plan was re-cut: the headline moves from a VS Code review cockpit to the bot's PR comment, and VS Code is re-aimed at the author, not the reviewer.

Revised spine — build order

  1. Land Slice 2.A — finish the diff engine that powers everything (already built).
  2. In-PR visual diff comment (new) — the bot posts a rendered before→after in its own PR comment. New top priority: cheapest, biggest reach, GitHub-native, and the one thing GitHub can't do itself.
  3. Slice 6 — Config UX (promoted) — the control plane for what the bot touches and what triggers creation.
  4. Validation gate — instrument and let engagement decide whether the heavy slices earn their weeks.
  5. Author track — Slice 7 (pre-emptive local sync) + Slice 5 (edit mode) + create-on-commit nudge (new): where the IDE genuinely wins.
  6. Build the gated cockpit as a bet — Slice 3 is the only login-gated surface (the activation lever) and is already in motion, so finish it ahead of the data and instrument it. Slice 2.6 (shared component) is unblocked once it ships. Only Slice 4's heavy rename/restructure UI still waits for a signal.
Why we're building the cockpit ahead of the data. We don't yet have data confirming demand for the review cockpit — and we're building Slice 3 anyway, deliberately. Its job is activation, not review volume: it's the only login-gated surface, a concrete reason to create an account, which matters because 96% of activated users never log in. It's also already in motion. And the new top-of-funnel work (in-PR diff, create-on-commit) plus the runaway creation commands should grow the user base and the GitHub App's reach — feeding the pool that could ever use review. That's the flywheel. The discipline that keeps it a measured bet, not a hope: instrument the gate for logins driven (not just reviews), and pair the cockpit with the in-PR diff so there's an actual path into it.

05

The review surface

Two altitudes: one diagram, or the whole branch.

The experience has two levels that share one renderer. At ground level is the single-diagram surface (Slice 2): banner, source diff, visual preview with the change highlighted, and Accept / Reject / Edit. Above it is the cockpit (Slice 3): a sidebar that lists every bot-edited diagram so you never have to hunt through the Explorer.

06

The login gate — decided

Gate the aggregation and the bulk action. Never the review itself.

The sharpest product decision in the project is where the login wall sits. The answer: the same gate the AI / Diagram Suggestions panel already uses, applied only to the multi-diagram list and Accept all.

CapabilityLogged outLogged in
Open & review one bot-edited diagram (banner, diff, Accept/Reject/Edit)✓ Free✓
Multi-diagram "Review Mermaid Sync" listSign-in prompt✓
Accept all (bulk resolve)Locked✓
Why this shape. The moment of value — actually reviewing a change — must never carry friction. But the cockpit is genuinely more valuable when you're triaging a big PR, which is exactly the moment a developer will trade an account for convenience. Login lift without a paywall.
Design note

"Same pixels, two states. The signed-out sidebar isn't empty or hidden — it shows a calm sign-in prompt exactly where the list would be, with the per-diagram path still open underneath. We sell the upgrade without blocking the work."

07

Evolution & review rounds

How the thinking moved — and where feedback overruled the first plan.

Short version: shipped 0–2, learned Slice 2 was "heavy to use," drafted the enhanced diff UX on a branch, borrowed the editor's node-highlight, and adopted Prashant's sidebar for Slice 3.

The full paper trail (7 rounds)
Slices 0–2
Acknowledge → diff → act, shipped

Bot-edit signal, the auto-regenerated banner + tab dot, then the single-diagram visual diff with Accept / Reject / Edit. The thesis-proving sequence landed.

Slice 2 feedback
"Heavy to use"

The single review surface tested as heavy. That soft-blocks Slice 3 — stacking N heavy surfaces compounds the load — and makes a progressive-disclosure pass a prerequisite, not a polish item.

Wilson feedback + mockup
Removals: list, not ghosts

The original "ghost treatment" (a faded removed node drawn on the diagram) was dropped in favour of a removed-node list — ghosting on a re-laid-out diagram is unreliable. Carried into Slice 4 as the current lean.

feature/pr-review
Enhanced diff UX drafted

~815 lines: AST summary banner, collapsible change list, click-to-focus, role-aware highlighting, line-diff fallback. The commit cb03e18 ("visual diff preview, changes chrome, beside code panel") is the core of it. Built, not yet landed (§10).

From the editor team
AI node highlight, reused

The unified editor's progressive-onboarding work produced a framework-agnostic node-highlight (outline a node's own shape in the brand gradient). It folds into the diff as a role-aware highlighter; the companion generation-skeleton is parked for Slice 7's local sync.

Slice 3 kickoff
Prashant's sidebar proposal

A dedicated Review Mermaid Sync sidebar listing every bot-modified diagram, click-to-open, per-diagram review status, and an optional git stash-style aggregated view — placed like the Improve Diagram panel. Became the Slice 3 direction.

The one-line read. The zero-setup, in-editor commands are thriving, and the GitHub App is one of our most-used systems. The VS Code review experience is the only piece almost no real user has touched yet — and the reason isn't the one we assumed.

What leadership should take away

  • In-editor visibility drives adoption. The three runaway features — Generate from Code, Open Chat, Improve Diagram — are all zero-setup CodeLens actions surfaced in the document. Improve Diagram, four days old, is already at ~1,600 uses from 665 users and climbing daily.
  • "Users don't know about the GitHub App" does not hold. The bot regenerates diagrams ~8,200 times a week in production (a new high). Awareness is high; promoting the workflow harder targets a gap that isn't there.
  • The gap is location, not awareness. Thousands of bot edits happen in GitHub PRs, where developers already review and merge. The VS Code review surface asks them to go somewhere else to re-review. That friction — not awareness — is what's suppressing usage.
The reframe

"We don't need blog posts about the GitHub App — people use it constantly. We need to meet the bot edits where they already are, and give review the same in-editor visibility that made the other commands take off."

Why the awareness hypothesis doesn't hold

If low review usage were an awareness problem, we'd expect the GitHub App itself to be under-used. The opposite is true: GitHub Regenerate now fires ~8,200 times a week and has climbed steadily for months — one of the most-used systems we have. Developers clearly know the workflow and rely on it.

Yet Review Mermaid Sync shows just two distinct users. So the issue isn't that people haven't heard of the bot — it's that the bot's edits land in GitHub PRs, where review and merge already happen, while our review surface lives in VS Code, which asks for an extra trip. Promotion can't close a location gap.

What to do instead
  • Route the existing regenerations into review. A deep link from a bot-edited PR/commit straight into the VS Code review, instead of expecting users to find it.
  • Give review CodeLens-grade visibility. Surface "this diagram was bot-regenerated — review it" in-editor on open. Slice 1's banner and Slice 3's sidebar already do exactly this — the roadmap addresses the real gap better than promo would.
  • Stand up the funnel before any spend. GitHub Regenerate → file opened in VS Code → Bot Edit Detected → Review Triggered → Accepted. We can't justify promotion against a funnel we can't yet see.
  • Consider surfacing review where the edits live (the GitHub PR), not only in VS Code — now the top-priority in-PR diff comment.
Data caveats — read before quoting
  • GitHub Regenerate's "unique users" equals its total — a server-side identity quirk. Treat the total as the real signal, not reach.
  • The review events are days old and concentrated in 2 distinct IDs (likely internal). Not enough real-user data to conclude anything beyond "too early."
  • Bot Edit Detected was instrumented on 2026‑06‑19 — its near-zero count partly reflects brand-new tracking, not just low use.
  • Improve Diagram has been live since Jun 15, not "yesterday." The combined headliner volume is ~82k events (Generate ~45k + Open Chat 37k), larger than the ~40k first cited.
09

Key decisions

What we chose, and why.

Five calls: working-tree scan · sidebar TreeView · batch accept, per-diagram reject · list (not ghost) · extract shared component after the 2nd consumer

Detection = working-tree scan (default)

Find bot-edited diagrams by scanning the branch's changed files for the bot trailer, rather than walking the full commit range. Simpler, holds the ~2-week estimate, and isolated behind one function so swapping to a commit-range walk later is a one-file change.

A sidebar TreeView, not an editor-area list

Per the mockup and Prashant's proposal, the cockpit is a dedicated VS Code TreeView placed like the Improve Diagram panel — not a tab in the editor area. The two are quite different to build; the sidebar won.

Accept all is a batch; reject stays per-diagram

"Accept all" resolves the listed set in one action. There is deliberately no "reject all" — rejecting is a per-diagram judgement, and a bulk reject is the kind of action people regret.

Removals: list, not ghost

Removed nodes are listed, not drawn as ghosts on a re-laid-out canvas. Reconcile to one treatment, not both (Slice 4).

Extract the shared component only after the second consumer exists

The shared diff renderer (2.6) waits for Slice 3. The right API shape is only knowable with two real consumers under real design pressure — doing it earlier means guessing.

10

Open questions

Still needs a call.

  • Where does "Accept" happen — VS Code or GitHub? With the in-PR diff comment now top priority, the accept action may belong on the PR itself, not only in the editor. Needs a call (see the User journey view).
  • Role-aware canvas colors. The diff highlight currently uses one pink→purple gradient for every changed node, so added and changed look identical on the diagram — the list pills carry the role, the canvas doesn't. Folded into Slice 2.6, but the color language needs deciding.
  • Rename threshold. How aggressive should rename heuristics be? A false "rename" is worse than a miss — needs tuning against real before/after examples (Slice 4).
  • Restructure detection. Does it reuse the Slice 2 node diff or need a richer structural diff? Ties into the shared-component work.
  • Single-diagram fallback. Confirm a one-diagram PR skips straight to the Slice 2 surface with zero list friction.
